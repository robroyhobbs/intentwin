## Product Overview

IntentBid is an AI-powered proposal generation SaaS using Intent-Driven Development (IDD).

**Tech Stack:**

- Next.js 16 with React 19
- Supabase (PostgreSQL + Auth + Storage)
- Google Gemini (gemini-3.1-pro-preview) for proposal generation
- Voyage AI for embeddings
- Stripe for billing (checkout, webhooks, customer portal)

**Target:** $10K MRR in 90 days

## Core Patterns

### IDD 3-Layer Model

Always respect the Intent-Driven Development architecture:

- **L1 (Company Truth)**: Canonical data - brand, capabilities, case studies. Locked, admin-only updates.
- **L2 (Proposal Intent)**: Client outcomes, win strategy, constraints. Human-reviewed.
- **L3 (Generated Content)**: AI-generated sections with claim traceability. AI executes, human verifies.

### Multi-Tenancy (Critical)

- All queries MUST scope by `organization_id`
- Never expose data across organizations
- RLS policies enforce isolation at database level

### Usage Tracking

- Track proposals created per billing period
- Track AI tokens consumed
- Enforce plan limits before generation

## API Route Patterns

**CRITICAL: Always use getUserContext for organization scoping!**

```typescript
import { getUserContext, verifyProposalAccess } from "@/lib/supabase/auth-api";

// Get user context with organization
const context = await getUserContext(request);
if (!context) {
  return NextResponse.json({ error: "Unauthorized" }, { status: 401 });
}

// For routes that access resources by ID, ALWAYS verify org access
const proposal = await verifyProposalAccess(context, proposalId);
if (!proposal) {
  return NextResponse.json({ error: "Not found" }, { status: 404 });
}

// For new queries, always filter by organization_id
const { data } = await adminClient
  .from("proposals")
  .select("*")
  .eq("organization_id", context.organizationId);
```

**Available helpers in `@/lib/supabase/auth-api`:**

- `getUserContext(request)` - Returns user + organizationId + role
- `verifyProposalAccess(context, id)` - Checks proposal belongs to user's org
- `verifyDocumentAccess(context, id)` - Checks document belongs to user's org
- `checkPlanLimit(orgId, key)` - Checks if org is within usage limits
- `incrementUsage(orgId, key, amount)` - Tracks usage

### 2026-03-06 - Gemini SDK Migration (`@google/generative-ai` → `@google/genai`), Client Centralization

- **New Gemini SDK (`@google/genai`) has a fundamentally different API surface**: Migrated from `GoogleGenerativeAI` to `GoogleGenAI`. Key differences: (1) `client.getGenerativeModel({model, generationConfig}).generateContent(prompt)` → `client.models.generateContent({model, contents, config})`, (2) response is returned directly (no `.response` wrapper), (3) `.text()` method → `.text` property, (4) `generationConfig` → `config`, (5) `systemInstruction` moves into `config`. Pattern: when migrating SDKs, map every API call site — the new SDK may flatten, rename, or restructure the entire call chain
- **Centralize AI client construction into a single exported factory**: Before migration, 4 files each created their own `GoogleGenerativeAI` instance with duplicated API key checks. Consolidated into a single `getClient()` export from `@/lib/ai/gemini.ts`. All consumers (`diagram-generator.ts`, `gemini-review-client.ts`, `mermaid-to-svg.ts`, diagram route) now import from one place. Pattern: when migrating an SDK, centralize client creation first — it reduces the migration surface and ensures consistent configuration (API keys, base URLs, options) across all call sites
- **Helicone proxy for Gemini removed — new SDK doesn't support `baseUrl`/`customHeaders`**: The old `@google/generative-ai` SDK accepted `RequestOptions` with `baseUrl` and `customHeaders` for Helicone proxy routing. The new `@google/genai` SDK uses a different configuration model that doesn't expose these options. Removed `geminiHeliconeOptions()` entirely and its tests. Voyage AI Helicone proxy (raw fetch) is unaffected. Pattern: observability proxies that depend on SDK-specific request hooks may break on SDK upgrades — verify proxy compatibility before migrating, and have a plan for alternatives (e.g., OpenTelemetry, direct API logging)
- **`GOOGLE_API_KEY` is the only required env var**: The new `@google/genai` SDK uses `GOOGLE_API_KEY`. The legacy `GEMINI_API_KEY` fallback was removed after confirming production stability. Pattern: when migrating env vars, keep the old name as a temporary fallback, verify production works, then remove the old var to avoid confusion
- **Image generation `responseModalities` values changed**: Old SDK used lowercase `["image", "text"]` (with a `@ts-expect-error` to suppress type errors). New SDK uses uppercase `["IMAGE", "TEXT"]` as proper enum values, eliminating the need for `@ts-expect-error` comments. Pattern: SDK migrations often fix type-safety issues — remove suppression comments (`@ts-expect-error`, `as any`) and check if the new SDK provides proper types
- **`inlineData` null safety**: Diagram generator checked `if (part.inlineData)` but the new SDK can return `inlineData` with a null `data` field. Changed to `if (part.inlineData?.data)` to prevent passing null base64 data downstream. Pattern: when migrating SDKs, add defensive checks on response shapes — even "equivalent" responses may have subtle nullability differences

### 2026-03-05 - Pipeline Reliability Hardening, Race Condition Fixes, Graceful Degradation

- **`Promise.allSettled` prevents one failed fetch from killing an entire pipeline stage**: Bid scoring fetches (evidence, requirements, criteria) were using `Promise.all` — one network error or timeout caused the entire scoring to fail. Switched to `Promise.allSettled` so partial data degrades gracefully (e.g., scoring proceeds with 2/3 data sources). Pattern: any pipeline stage that fetches multiple independent data sources should use `Promise.allSettled` and degrade on partial results rather than fail entirely
- **Silent `null` returns from DB queries mask errors — throw instead**: `fetchL1Context` returned empty arrays on both "no data exists" and "Supabase query error" cases. Downstream code couldn't distinguish between a proposal with no team members vs a database outage. Fix: check `.error` on Supabase responses and throw, reserving empty arrays for genuinely empty results. Pattern: data-fetching functions should distinguish "no results" (return empty) from "query failed" (throw) — silent empty returns on errors cause cascading wrong behavior
- **Finalize race condition: poll before declaring failure**: The finalize route was checking section statuses once and immediately marking non-`completed` sections as `failed`. But sections could still be generating (the client fires section requests, then calls finalize). Added `waitForTerminalState` that polls up to 30s for all sections to reach terminal status before finalizing. Pattern: when an orchestration step depends on prior async steps completing, poll with backoff rather than snapshot-and-judge — the snapshot may catch in-flight work
- **Propagate degradation warnings through the full pipeline to the UI**: When L1 context is partially missing (e.g., no team members, no evidence), the pipeline silently continued with degraded output quality. Added a `warnings[]` array to pipeline context that accumulates issues during context build, returns them in the setup API response, and surfaces them as an amber banner in the strategy phase UI. Pattern: pipeline degradation should be visible to users — accumulate warnings at each stage and surface them in the UI, don't just log server-side
- **Sequential retry prevents 429 rate-limit bursts**: The "Retry All Failed" button was firing all section regenerations concurrently via `Promise.all`, triggering Gemini 429 errors. Changed to sequential `for...of` with `await` per section. Pattern: retry-all UI actions should execute sequentially by default — the user clicked "retry" because something failed, and hammering the API concurrently will likely fail again for the same reason
- **Clear stale error state after successful retry**: After a section was successfully regenerated, the proposal-level `generation_error` field still showed the old error message. Users saw "Failed" banners even though all sections were now complete. Fix: clear `generation_error` in the regenerate route on success. Pattern: when a retry succeeds, clear error state at every level that cached the original failure — check both the item-level and parent-level error fields
- **`NextResponse.rewrite()` doesn't work in route handlers**: Attempted to use `NextResponse.rewrite()` to redirect the deprecated monolithic `/generate` route to the new `/generate/setup` route. It silently failed (returned empty response). Route handler rewrites are not supported in Next.js App Router — use `NextResponse.redirect()` or return an explicit error. Replaced with a 410 Gone response directing callers to the new routes. Pattern: when deprecating an API route, return a 4xx with a clear message pointing to the replacement — don't try to transparently proxy
- **`source_text` truncation causes "insufficient data" scoring**: Bid scoring returned "insufficient data" for most factors because `source_text` was truncated to 1,000 chars during extraction — too little for the AI to score against. Increased capture to 10K chars and changed the scoring prompt to always include source text (8K when structured data is sparse, 3K when plentiful). Pattern: when AI returns "insufficient data," check the input pipeline for truncation before adjusting the prompt — the model may literally not have enough text to work with
- **Route-level integration tests catch orchestration bugs that unit tests miss**: Added 13 tests covering setup/section/finalize route behaviors: feature gate checks, token limit validation, idempotency guards, error handling, and finalize polling. These caught the finalize race condition and the rewrite bug that unit tests of individual functions would have missed. Pattern: for multi-route orchestration flows, write route-level integration tests that exercise the full request/response cycle — unit tests on helpers don't catch route wiring issues

### 2026-03-04 - Client-Orchestrated Generation, Inngest Removal, Bid Scoring Resilience, Vercel Build Fixes

- **Client-orchestrated generation bypasses Vercel 60s timeout**: The Inngest-orchestrated generation hit Vercel Hobby plan's 60s function timeout. Rewrote to 3 client-driven sequential API calls: `/generate/setup` (build pipeline context, create section rows), `/generate/section` (one section per request, <60s each), `/generate/finalize` (count results, trigger quality review). Browser drives the loop, so no single request exceeds the timeout. Key safeguards: server-side idempotency (duplicate section guard), orphan recovery for stale proposals (>15 min), browser refresh resumability via `resumeDraftFlow()`, and `GenerationNonRetriableError` to prevent token-burning retries. Pattern: when serverless timeouts cap your orchestration, move the loop to the client with per-step API calls — each step must be idempotent and resumable
- **Removing Inngest from generation requires careful dead-code cleanup**: After client orchestration replaced Inngest for generation, dead Inngest code persisted: `regenerateSectionFn` in the registry (never triggered), `section/regenerate.requested` event type, and the old monolithic `generate-single-section.ts` under `inngest/functions/`. Regeneration route was still dispatching Inngest events instead of calling `generateSingleSection` directly. Cleanup spanned 3 commits (560b07f → 7bd4dd6). Pattern: when replacing an orchestration layer, grep for all event names and function references — dead dispatchers that call removed functions fail silently
- **`after()` from Next.js enables non-blocking background work on Vercel**: Regeneration route needed to return immediately while generation continued in the background. `after()` (from `next/server`) queues work after the response is sent — replaces Inngest for simple fire-and-forget tasks. Gotcha: `after()` still runs within the route's `maxDuration`, so set `maxDuration = 300` on routes that use it for AI generation
- **Bid scoring needs unwrapScoreData() for nested JSON envelopes**: Gemini sometimes wraps scoring responses in an extra JSON envelope (`{ data: { factors: [...] } }` instead of `{ factors: [...] }`). `unwrapScoreData()` recursively checks for nested structures containing the expected shape. Also added `mergeScores()` that only replaces failed scoring factors on retry, preserving previously successful scores. Pattern: AI response parsers should handle at least one level of unexpected nesting
- **Partial bid score failures need granular retry, not all-or-nothing**: Bid scoring can partially fail — some factors parse, others return "insufficient data" or "could not be scored" fallback markers. Client-side now auto-retries when >= half the factors are fallback scores, and `mergeScores()` only overwrites the failed factors. Pattern: any AI scoring with multiple independent factors should support partial retry with selective merging
- **DB column mismatches cause silent generation failures**: `proposal_sections` table lacked a `metadata` column in production. The DB update included metadata alongside `generated_content`, so the entire UPDATE failed — content was never saved, sections showed "failed" with no error. Fix: split metadata write into a separate non-blocking call (34eddf1). Also added migration 00007. Pattern: when adding a new column to an existing UPDATE statement, the UPDATE must either (a) confirm the column exists first or (b) write the new column in a separate call so the critical write succeeds
- **Error surfacing requires tracing through the full pipeline**: Section generation errors were swallowed at three levels: (1) `generateSingleSection` didn't check the DB write result, (2) setup route didn't verify `generation_metadata` write success, (3) draft UI showed "Failed" with no error message. Fixed across 4 commits (f447b3d → d1fcd67 → bb94c6a → 48b6d1a). Pattern: after fixing a generation bug, trace error messages from AI response → DB write → API response → client display — verify each hop preserves the error
- **Vercel static build fails on pages using Supabase client-side**: Settings pages (brand-voice, branding, company), auth pages, and onboarding use Supabase client which requires runtime env vars. During Vercel build, these are unavailable, causing static pre-render failures. Fix: add `export const dynamic = "force-dynamic"` to each page. Required 3 progressive commits as pages surfaced one-by-one (b53dcc4 → 6004edd → bb0ce9d). Pattern: any page that calls `createBrowserClient()` or reads cookies at render time needs `force-dynamic` — audit all pages after adding new Supabase calls
- **Codex audit findings improve edge-case resilience**: External audit (Codex) found 4 issues in one pass: (1) setup route should check token availability before claiming the proposal (prevents stuck GENERATING on insufficient tokens), (2) zero-section guard missing in both run and resume flows, (3) AI blocked/timeout errors should produce fallback placeholder content instead of failing entirely, (4) `<thought_process>` prompt tag caused model refusal/blocks — replaced with internal planning guidance. Pattern: periodic external code audits catch edge cases that iterative development misses — schedule them after major architecture changes
- **`<thought_process>` tags in prompts increase Gemini refusal rates**: The editorial standards prompt included `<thought_process>` XML tags for internal reasoning. Gemini interpreted these as instructions to expose chain-of-thought, triggering safety filters on some inputs. Replacing with natural-language planning guidance ("Before writing, internally consider...") eliminated the refusals. Pattern: avoid XML-style tags in AI prompts that resemble system/meta instructions — use natural language for reasoning guidance

### 2026-03-02 - Gemini Rate Limit Resilience, Parallel Extraction, Decision Coach Redesign, Export Placeholder Resolution

- **Gemini rate limits require NonRetriableError wrapping, not batching**: The rate limit saga spanned 5 commits (f31b130 → 773fcc8). First attempt: batch 3 sections at a time with delays. Second: stagger parallel calls 300ms apart. Final solution (4f7b641): fire ALL sections in true parallel via `Promise.allSettled`, but wrap timeout and `SAFETY` errors in Inngest `NonRetriableError` so hopeless calls aren't retried. Per-section timeout reduced from 180s→90s (most complete in 8-15s). Old approach: 2-12 min. New approach: ~30s total. Pattern: don't throttle to avoid rate limits — let the queue handle concurrency and make non-transient errors non-retriable. Only 429/503 deserve retries
- **Validate Gemini `response.candidates` before calling `response.text()`**: Safety filter blocks, empty responses, and content filtering cause cryptic SDK errors when you call `response.text()` on an invalid response. Fix (76ba90b): check `response.candidates` existence and `finishReason` before extraction. `AI_BLOCKED` errors trigger model fallback (flash may succeed where pro is blocked) and are flagged non-retriable to stop token burn. Pattern: every `generateText()` call needs a response validator — the SDK doesn't throw on safety blocks, it returns a response with no usable text
- **Parallel micro-extractors beat monolithic RFP extraction 3x**: Split single 30-60s Gemini call into 3 focused parallel extractors (e990d09): quick fields (truncated to 30K chars, ~5s), requirements (full content, ~10s), structure+gaps (full content, ~10s). Wall time drops to `max(calls)` = ~10-20s. Uses `Promise.allSettled` so partial results are usable if 1-2 extractors fail. Falls back to original single-call extraction if all 3 fail. Pattern: when an AI call takes >20s, look for independent sub-tasks that can run in parallel — the same content can be parsed for different concerns concurrently
- **`maxOutputTokens` increased to 8192 for section generation (5th occurrence)**: Large prompts with L1 context + persuasion layers left insufficient output budget at 4096 tokens. This is the fifth time `maxOutputTokens` has been the root cause of truncated or broken output (see 2026-02-22, 2026-02-20). Rule: `maxOutputTokens = 2x estimated output` is now validated — any section generation call should use 8192 minimum
- **Per-section retry UI for partial generation failures**: When some sections succeed and others fail, users shouldn't regenerate the entire proposal. Draft phase now detects partial success, shows failed sections with retry buttons, and calls the existing per-section regeneration endpoint (79094f3). Pattern: any batch operation that can partially fail needs a partial-retry UI, not just all-or-nothing
- **Decision Coach converged on verdict/gaps/strengths after 6 iterations**: The coach went through: progress ring + timeline + tips (4eae7e0) → deep data surfacing (5fb2585) → UX feedback simplification (78db002) → finalize redesign (5886365) → RadarChart addition then removal (afebc45, 683f412) → final verdict/gaps/strengths layout (84254ec). Key lesson: the RadarChart looked impressive but added visual complexity without actionable insight — verdict + gaps + strengths provides the same information with clearer next-steps. Pattern: advisory panels should surface "what to do next," not "how the data looks." If a visualization doesn't directly drive a user action, remove it
- **sessionStorage persistence prevents lost wizard progress on refresh**: State serialized to sessionStorage on every reducer dispatch (f317818). Key gotchas: `Set` must be converted to array for JSON, `File[]` cleared on restore (can't serialize), transient flags (`isExtracting`, `generationStatus === "generating"`) reset to safe states on load. `generationStatus` resets from "generating" to "failed" because polling won't auto-resume after navigation (f373489). Pattern: when persisting reducer state, classify each field as persistent (data), transient (UI flags), or non-serializable (Files, Sets) and handle each category explicitly
- **Export placeholders resolve from L1 sources at export time**: `{signatory_name}`, `{signatory_title}` resolve from L1 team members (first registered member), falling back to user profile then org name. `[CASE STUDY NEEDED: ...]` markers fuzzy-match against evidence library by industry/scope keywords. `[TEAM MEMBER NEEDED: ...]` matches against team members by role/skills. Only strips markers when no L1 data exists (7e4b9c5, 3eb8ab9). Pattern: AI-generated content should use placeholder markers, not hallucinated data — resolve placeholders at export time from authoritative L1 sources
- **Bid scoring determinism requires 5 controls**: Temperature 0 (was 0.3), explicit score bands per factor, rounding to nearest 5, alphabetically sorted inputs (evidence, requirements, criteria), and evidence-citing rules that prevent hallucinated rationale variation (cabf7a8). Pattern: when AI scoring must be reproducible for demos, control all sources of randomness — temperature, input ordering, output quantization, and citation constraints
- **Client-side polling timeout must match server `maxDuration`**: Document processing polling timed out at 60s client-side while server had `maxDuration = 300`. Large files need 2-3 minutes for PDF parsing + embedding generation via Inngest (e9fabb8). Pattern: when client polls a server-side background job, set `clientTimeout >= serverMaxDuration` — otherwise the client gives up while the server is still working
- **Triple error-surfacing bug in document processing**: Three independent bugs prevented users from seeing extraction errors (6b7ecea): (1) `DocxParseError` wasn't handled in Inngest worker (only `PdfParseError`), (2) client parsed `body.message` but API returns `body.error`, (3) polling showed generic "failed on server" instead of `processing_error` from DB. Pattern: trace error messages through the full pipeline (worker → DB → API → client) to verify they actually reach the user
- **API response shape mismatches cause silent failures**: Export response parsed as `{ data: { downloadUrl } }` but API returned `{ downloadUrl }` directly (9021b11). camelCase vs snake_case mismatch in another case (0370be6). Pattern: when connecting client to API, read the actual route handler response shape — don't assume nesting or casing conventions
- **Zero-chunk documents need pre-extraction validation**: Image-heavy DOCX files process successfully but yield no extractable text chunks (326770e). Without detection, extraction proceeds with empty input and produces garbage. Pattern: validate that document processing produced usable output (chunk count > 0) before proceeding to AI extraction

### 2026-03-01 - Create Flow (4-Phase Workspace), Auth Fetch Threading, Decision Coach

- **Thread `authFetch` through helper modules — plain `fetch()` silently drops auth headers**: The entire create flow (intake, strategy, draft, finalize) was built with plain `fetch()` calls in helper files. Every API call returned 401 because the `Authorization` header was missing. Fix (3175241): define `type FetchFn = (url: string, options?: RequestInit) => Promise<Response>` and pass the `useAuthFetch()` return value from React components down into all helper functions. 8 files changed, 152 insertions. Pattern: when splitting API-calling logic into helper modules outside React components, the auth fetch function must be dependency-injected — helpers can't call hooks directly
- **Reducer-first development works for multi-phase flows**: The create flow was built bottom-up: types (`create-types.ts`, 107 lines) → reducer (`create-reducer.ts`, 156 lines) → tests (111 lines, 8 test cases) → provider → shell → phases. Having the reducer tested before any UI existed meant phase transitions, confidence scoring, blocker resolution, and section review state all worked correctly from the start. Same pattern as the wizard rewrite (2026-02-24). Pattern: for multi-phase UIs, the development order should be types → reducer → reducer tests → provider → shell → phase components
- **Rule-based Decision Coach avoids AI latency on every phase change**: The `coach-content.ts` (300 lines) generates contextual guidance per phase using pure functions over `CreateFlowState` — no AI calls. It computes risk flags from extraction gaps, bid score thresholds from `SCORING_FACTORS`, section generation status, and unresolved blockers. The coach updates instantly on every state change. Pattern: for real-time advisory panels, prefer rule-based content derived from existing state over AI-generated suggestions — save AI calls for content generation, not navigation guidance
- **Shared UI atoms (`_components/shared/`) prevent duplication across phases**: Created 5 shared components (blocker-item, citation-pill, confidence-pill, risk-flag, section-card) used across draft and finalize phases. Total: 385 lines of reusable UI. Without extracting these early, each phase would have reimplemented badge/pill rendering inline. Pattern: when building a multi-phase flow, identify shared visual atoms during the types phase and build them before any phase component
- **`_components/` co-location scales better than flat directories**: The create flow has 24 files totaling ~3,300 lines, organized as `create/_components/` (root-level orchestration), `phases/` (phase-specific logic), and `shared/` (reusable atoms). This mirrors the `proposals/new/_components/` pattern. The `_` prefix prevents Next.js from treating the directory as a route segment. Pattern: for complex page features, use `page/_components/{phases,shared}/` — it keeps related files co-located without polluting the route tree
- **Confidence scoring as a derived value prevents state drift**: `createReducer` computes `confidence` on every action by counting completed phases (25% each) minus penalties for unresolved blockers (5% each) and unreviewed sections (2% each). It's never set directly — always derived from other state. This eliminated a class of bugs where confidence could become stale. Pattern: scores and percentages that depend on multiple state fields should be computed in the reducer, not stored independently
- **Exponential backoff for generation polling prevents thundering herd**: `draft-helpers.ts` polls section generation with `POLL_INITIAL_MS = 2000`, `POLL_BACKOFF = 1.5`, `POLL_MAX_MS = 10000`, and a 10-minute timeout. Without backoff, 10 concurrent section generations would hammer the API at 500ms intervals. Pattern: any client-side polling loop needs three params — initial interval, backoff multiplier, and max interval — plus an absolute timeout to prevent infinite loops
- **Error boundary at the route level catches phase rendering crashes**: Added `create/error.tsx` (43 lines) to catch React rendering errors in any phase component. Without this, a crash in the finalize phase (e.g., undefined blocker data) would white-screen the entire dashboard. Pattern: every new route directory with complex state should include an `error.tsx` boundary — it's 40 lines of boilerplate that prevents cascading UI failures

### 2026-02-28 - Intelligence Contracts, Server-Timing Observability, Wizard State Extraction, Rollback Drills

- **Interface-first refactoring enables testability**: Added `IntelligenceService` interface (`contracts.ts`) and made `IntelligenceClient` implement it. All consumers now depend on the interface, not the concrete class. Also exported a `getIntelligenceClient()` factory alongside the singleton. Pattern: when refactoring a module that external code depends on, add an interface first, then refactor implementations behind it — this makes mock/stub injection trivial for tests
- **Server-Timing headers give free browser observability**: Added `Server-Timing` metrics to intelligence, generate, and export API routes (wave 3). Each route now reports stage durations (fetch, generate, upload, sign) in the response header. Browser DevTools Network tab shows these automatically — no APM vendor required. Pattern: for API routes with multiple async stages, add `Server-Timing` headers with `formatTimingMs()` — it costs 2 lines per stage and gives instant production diagnostics
- **Parallel DB fetches reduce export P99 latency**: Export route was fetching proposal then sections sequentially. Wrapped in `Promise.all` to run both queries concurrently. Same pattern applied to the intelligence route. Pattern: when an API handler makes 2+ independent DB queries, always `Promise.all` them — the cost is trivial and the latency improvement is the slower query's duration
- **Map caches leak memory without eviction**: Intelligence client's `Map<string, CachedEntry>` cache had no size limit or cleanup. In long-lived serverless instances (Vercel keeps warm for ~15min), the cache grew unbounded. Added `CACHE_MAX_ENTRIES` (250), periodic expired-entry cleanup every 25 writes, and LRU overflow eviction. Pattern: any in-memory `Map` cache needs three things — max size, expiry cleanup, and overflow eviction. Without all three, it's a memory leak with a TTL delay
- **Wizard state belongs in the domain layer, not React components**: Extracted `WizardState`, `WizardAction`, and `wizardReducer` from `wizard-reducer.ts` (React component directory) into `lib/proposal-core/wizard-state.ts`. Also extracted draft persistence into `wizard-draft.ts`. The wizard components are now thin UI shells around domain-layer state. Pattern: if a reducer has 200+ lines and 15+ action types, it's domain logic wearing a React costume — extract it to `lib/` so it's testable without rendering
- **Release gate workflows need secret validation upfront**: The `release-gate.yml` workflow took 4 commits to stabilize (3868e30 → c5753cf → 53d0fc4 → 4f8ef68) because deploy command secrets were validated too late — empty strings, missing fallbacks, and Vercel token edge cases all surfaced at deploy time. Fix: added explicit validation step that checks `DEPLOY_COMMAND` presence before attempting deploy. Pattern: CI workflows that consume secrets should have a dedicated validation step at the top — failing early with a clear message beats failing mid-deploy with a cryptic error
- **Monthly rollback drills catch broken recovery before incidents do**: Added `rollback-drill.yml` — a monthly cron workflow that runs pre-drill smoke, triggers a rollback webhook, then runs post-drill smoke. ### 2026-02-25 - Gotenberg PDF, AI Stack Simplification, RFP-Driven Sections, Export Polish, Tiered Pricing

- **Gotenberg replaces Puppeteer/Chromium for PDF export**: After 5 commits fighting serverless Chromium binary management (`@sparticuz/chromium`, `chromium-min`, remote binary downloads), replaced the entire approach with Gotenberg — a Docker container running Chromium that accepts HTML via multipart POST and returns PDF. Production sends HTML to `GOTENBERG_URL`; development falls back to local Chrome via dynamic `puppeteer-core` import. Eliminates 50MB binary bundles, cold-start downloads, and `outputFileTracingIncludes` hacks. Pattern: when a native binary dependency causes recurring serverless issues, externalize it to a dedicated service. Deploy script: `scripts/deploy-gotenberg.sh` for Cloud Run
- **HTML-to-PDF needs explicit print overrides**: Gotenberg renders HTML in a real browser, but sections using CSS transitions/opacity animations rendered as invisible in the PDF. Fix: added a `forPdf` option to `generateHtml()` that injects `opacity: 1 !important; transform: none !important;` overrides, hides the interactive sidebar TOC, and forces `print-color-adjust: exact`. Pattern: any HTML designed for interactive viewing needs a separate CSS pass for print/PDF — animations, scroll triggers, and hover states all break in paged rendering
- **Multi-LLM quality council was complexity without proportional value**: Removed Groq (Llama 3.3) and Mistral from the 3-judge quality review council, leaving Gemini-only. The multi-model approach added latency (3 parallel API calls), required 3 API keys, and the score variance between models made aggregation unreliable. Pattern: start with one good model, only add multi-model voting if you have empirical evidence of quality improvement. Also extracted `fetch-l1-context.ts` from the bloated `context.ts` to keep files under 300 lines
- **RFP-driven custom sections are now first-class**: Added `RfpSectionRequirement` and `RfpEvaluationCriterion` types (`src/types/intake.ts`). Sections prefixed `custom_` bypass `SECTION_CONFIGS` lookup and build prompts dynamically from RFP-extracted metadata (title, description, specific requirements). The step-configure wizard now shows evaluation criteria with weights and maps them to sections. Pattern: when a fixed section taxonomy can't cover all RFP structures, support a `custom_` escape hatch with runtime prompt construction — but keep the same generation/review pipeline
- **User section selections were silently ignored**: The wizard Step 3 let users select/deselect sections, but `generate-proposal.ts` always generated all sections from `buildSectionList()`. Fix: filter `allSections` through `ctx.intakeData.selected_sections` before creating section rows. Fallback: if `selected_sections` is missing (legacy proposals), generate all. Pattern: trace user choices through the full pipeline (UI → API → Inngest → generation) — wizard state stored in React doesn't help if the backend never reads it
- **Tone modulation in editorial standards**: Added `tone` parameter to `buildEditorialStandards()` supporting 4 modes: professional (default, no modifier), conversational (contractions, direct address), technical (precision, specs, minimal marketing), executive (outcomes, ROI, short decisive sentences). Each mode injects a `## WRITING TONE` block into the section prompt. Pattern: tone is a prompt-engineering concern, not a post-processing concern — inject it at generation time, not via editing afterward
- **Static sources/ directory was bleeding into production**: The `sources/` directory contains Capgemini demo content (case studies, methodologies) used during development. `buildPipelineContext()` loaded these unconditionally, so real customer proposals included fabricated Capgemini examples. Fix: gate `loadSources()` behind `process.env.NODE_ENV !== "production"`. In production, every org has its own L1 context in Supabase. Pattern: demo/seed data must be explicitly gated — file-based content that "just works" in dev becomes a data leak in production
- **Slide builder had hardcoded fallback content**: When section content was empty, the PPTX slide builder fell back to fabricated metrics ("95% Client Satisfaction", "40% Cost Reduction") and generic phase names. - **DOCX now embeds diagram images and has a TOC**: Extended the raw OOXML DOCX generator to fetch diagram images in parallel (with 8s timeout per image), embed them in `word/media/`, and add proper relationship entries. - **HTML export rebranded from `--cap-*` to `--brand-*` CSS vars**: All CSS custom properties renamed from `--cap-blue`, `--cap-dark` etc. to `--brand-primary`, `--brand-dark`. Pattern: when building multi-tenant SaaS, use generic token names from day one — renaming CSS vars across 500+ lines of styles is tedious
- **plan_tier CHECK constraint was missing values**: The DB constraint only allowed `trial`/`active`/`cancelled`. Adding `free` and `invite` required a migration (00005). Always audit CHECK constraints when adding new enum values to a column — a missing value causes hard 500s from Postgres

### 2026-02-15 - Race Conditions, Data Normalization & DevOps

- **Race condition between regenerate and quality review**: Fire-and-forget async operations (regenerate section, quality review) can run simultaneously, leaving sections stuck in `generating` state. - **ExtractedIntake nesting gotcha**: Copy-pasted RFP content arrives as a nested `ExtractedIntake` object (`.extracted.field.value` wrappers) but `buildRfpSummary()` expected flat top-level fields, producing 0/0 bid scores with no data.
